今天，我们终于发布了 **Antdv Next 的第一个版本 —— v1.0**。

在经历了一年多的酝酿与开发之后，我们完成了这一代 Ant Design 设计体系在 Vue 生态中的第一次完整落地。

## 为什么要开发 Antdv Next？

从个人角度来说，我一直非常喜欢 Ant Design。  

无论是在设计体系还是工程体验上，它都是我用过最顺手、最成熟、也最具审美一致性的组件库之一。

也正因为如此，在很早之前我们就在公司项目中选择了 antdv，并基于它做了大量二次封装。  

在 antdv v4 重构期间，我也有幸加入社区，参与了大约一年的开发与维护工作，并基于它构建了 [antdv-pro](https://github.com/antdv-pro/antdv-pro) 这一套管理后台基础框架。

但在这个过程中，我逐渐意识到一个现实问题：  

**基于传统 CSS 的主题切换方案，在性能和工程复杂度上已经越来越难以支撑大型项目。**  

与此同时，antdv 的版本迭代也逐渐放缓，这让我们很难继续在原有架构上推进更激进的设计系统能力。

这是我第一次认真思考：  

是否有必要从架构层面，重新实现一套真正面向“下一代设计系统”的 Vue 组件库？

第二个更重要的原因，是人。

在过去这一年里，有几位朋友和我抱着同样的想法，一起开始了 Antdv Next 的尝试。  

非常感谢他们在业余时间对 vue-components 的持续投入，如果没有他们的坚持，这个项目很可能无法走到今天。

特别感谢 [@selicens](https://github.com/selicens) 和 [@cc-hearts](https://github.com/cc-hearts)，  

他们从零开始参与组件设计与工程实现，一直坚持到现在，从未抱怨过任何“看不到尽头”的问题。  

也感谢 [@kovsu](https://github.com/kovsu)，虽然因为现实原因离开了项目，但我完全理解，一个长期投入却难以预期结果的项目，本身就是一件需要勇气的事情。

最后一个决定性因素，是 **Ant Design v6 的发布**。

v6 不再只是一次版本升级，而是一次设计系统层面的范式转变：  

默认 CSS Variables、Token 化设计、生成式静态样式、完整的样式运行时能力……  

这些能力第一次让“设计系统”真正成为一套可运行、可演进的工程基础设施。

那一刻我非常明确地意识到：  

**我们需要一个与 Ant Design v6 对齐的 Vue 实现。**  

不仅是 API 层面的兼容，更是设计理念与工程范式的同步。

这，正是 Antdv Next 最初诞生的原因。

## 设计调整

我们选择了要去做这件事，我们就要符合vue的设计范式，所以我们把一些属性在react的基础上做了一些调整，让他去`react`味儿！

前期我们能看到的全局通用的属性就是`className`，这个属性满满的都是react的味道，所以我们前期决定了把和这个相关的通用属性都改造一下：

* classNames -> classes
* rootClassName -> rootClass

这样我们就做到了第一步的去react味儿，当然我们对于antd下个版本就要删除的属性或者说是为了兼容antdv，我们并没有调整其他的原本就存在的属性，但是这些属性大概会在下个大版本，我们会完全移除掉，不再保留，所以我们也不推荐大家去使用原来的带着className的api了。
